# Backend Agent Steering

## Backend Goal

Build a clean NestJS REST API for authentication, profile access, and task management backed by PostgreSQL.

## Required Backend Scope

- User registration
- User login
- JWT authentication
- Authenticated profile endpoint
- Task create/read/update/delete endpoints
- Support task status management

## Stack Direction

- NestJS
- PostgreSQL
- JWT auth

No ORM is required for this project.

## Database Source Of Truth

Agents must treat [backend/scripts/setup-postgres.sql](/home/smriti/my-projects/Task-Manager/backend/scripts/setup-postgres.sql) as the current database source of truth when implementing backend APIs, DTOs, services, queries, and auth flows.

If an API shape uses friendlier names for frontend consumption, agents should still map those responses from the actual PostgreSQL column names defined in the script instead of inventing a different persistence schema.

## Current PostgreSQL Schema

The backend currently provisions:

- PostgreSQL role: `taskmanager_user`
- PostgreSQL database: `taskmanager_db`
- Table: `public.users`
- Table: `public.tasks`

### `public.users`

Use this schema when building registration, login, profile, and ownership logic:

- `id bigint` primary key generated by identity
- `first_name varchar(100)` not null
- `last_name varchar(100)` nullable
- `email varchar(255)` not null unique
- `password_hash text` not null
- `created_at timestamp` default `CURRENT_TIMESTAMP`
- `updated_at timestamp` default `CURRENT_TIMESTAMP`

Important implications:

- Authentication should query users by `email`
- Passwords must never be stored in plain text; only `password_hash` should be persisted
- Ownership relationships from tasks point to `users.id`

### `public.tasks`

Use this schema when building task CRUD and filtering logic:

- `task_id bigint` primary key generated by identity
- `task_status varchar(20)` not null default `todo`
- `task_creation_date timestamp` default `CURRENT_TIMESTAMP`
- `start_date_time timestamp` nullable
- `end_date_time timestamp` nullable
- `task_title varchar(255)` not null
- `task_desc text` nullable
- `created_by bigint` not null references `public.users(id)` with `ON DELETE CASCADE`

Allowed task status values are constrained in PostgreSQL to:

- `todo`
- `in_progress`
- `done`

Important implications:

- Task queries must be scoped by `created_by`
- Deleting a user cascades and deletes that user's tasks
- Backend code should preserve the stored status values exactly as defined above
- If response DTOs rename fields such as `task_id` to `id`, that mapping should be explicit and consistent

## Module Structure

Prefer a clean feature-based structure such as:

- `auth/`
- `users/`
- `tasks/`
- `common/`
- `database/`

Keep modules cohesive and avoid mixing unrelated concerns.

## API Design Rules

- Keep controllers thin
- Put business logic in services
- Use DTOs for request validation
- Use guards for auth-protected routes
- Return predictable JSON response shapes

## Authentication Guidance

Simple JWT authentication is acceptable for this project.

Minimum expectations:

- Register endpoint
- Login endpoint
- Password hashing
- JWT issuance on successful login
- Protected routes for profile and task actions

## Validation and Error Handling

This is a required evaluation area.

Agents must ensure:

- Request validation is enforced
- Invalid input returns clear 4xx responses
- Missing resources return proper 404 responses
- Unauthorized access returns proper 401 responses
- Forbidden actions return 403 where relevant
- Unhandled internal failures return safe 500 responses

Do not return raw stack traces, SQL errors, or framework internals to the client.

Prefer consistent error response shapes across the API.

## Task Domain Guidance

The database already uses task-specific column names. Prefer keeping persistence logic aligned with:

- `task_id`
- `task_title`
- `task_desc`
- `task_status`
- `task_creation_date`
- `start_date_time`
- `end_date_time`
- `created_by`

If the API exposes normalized response fields such as `id`, `title`, `description`, `status`, `createdAt`, and `userId`, document and centralize that mapping so it stays consistent across controllers and services.

## Authorization Guidance

- Users should only access their own profile and tasks
- Task queries must be scoped to the authenticated user
- Task update/delete actions must verify ownership

## Data and Persistence Guidance

- Use PostgreSQL as the source of truth
- Keep database access patterns consistent and explicit
- Keep schema naming clear and conventional
- Include a migration strategy
- Prefer explicit relations between users and tasks
- Check the SQL script before changing table or column assumptions
- Keep query code aligned with the actual column names in `setup-postgres.sql`

## Response Quality

API responses should be:

- Clean
- Consistent
- Easy for frontend consumption
- Free of unnecessary nesting unless justified

## Maintainability Rules

- Descriptive DTO, service, and method names
- Avoid giant service classes
- Extract shared concerns into `common/` only when truly shared
- Keep auth logic and task logic separate

## Suggested Backend Build Order

1. Database schema and user/task models
2. Auth module
3. User/profile endpoint
4. Task CRUD endpoints
5. Validation and error handling polish
6. Security and response cleanup
